Wireway — Information Security Policy

Effective date: June 27, 2026

This Information Security Policy describes how Jalil Muhammad (operating as Wireway) ("Wireway," "we," "us") protects the confidentiality, integrity, and availability of the data entrusted to Wireway Pro (the "Service"). It governs the people, processes, and technology used to operate the Service, and reflects the architecture and controls actually in place.

1. Purpose and Scope

Wireway Pro is a web and mobile application for independent electrical contractors. It helps contractors create estimates and quotes, schedule and cost jobs, track expenses and mileage, manage subcontractors, bill clients, and collect payment. This policy applies to all systems, data, code, and third-party services that make up the Service, and to everyone who operates or maintains it.

Where the Service accesses a user's financial information — such as bank and card transactions through Plaid — it does so only on behalf of, and at the direction of, the account holder, solely to provide features the account holder has requested. Wireway does not access, use, or process that data for any independent purpose.

2. Governance and Risk Management

The Founder serves as Wireway's Information Security Lead and holds overall accountability for security decisions, controls, and this policy. Security risks to the Service — including risks to confidentiality, integrity, and availability of contractor and consumer financial data — are identified, assessed, mitigated, and monitored on an ongoing basis as part of normal operations.

This policy and the controls it describes are reviewed at least annually, and additionally whenever a material change to the architecture occurs or following any security incident, so that the Service's safeguards keep pace with how it actually operates.

3. Identity and Access Management

Access to data follows the principle of least privilege. Application data is segregated per user through PostgreSQL Row-Level Security (RLS), so each account can read and write only its own records (owner-scoping). Every administrator and operator uses a unique, individual account — there are no shared logins.

  • Privileged keys — service-role and other privileged database keys are used only in server-side code; they are never exposed in the client application or browser bundle.
  • Multi-factor authentication — MFA is enforced on the core operational accounts that can change the Service or its data, including GitHub (source code), Vercel (hosting/deployment), and Supabase (authentication, database, and storage).
  • End-user authentication — end users sign in with email and password managed by our authentication provider. Consumer-facing multi-factor authentication is on the product roadmap.

4. Data Protection and Encryption

Data is encrypted in transit using TLS 1.2 or higher for all connections between users, the Service, and its subprocessors. Data at rest is encrypted through our managed PostgreSQL provider.

Plaid access tokens — the credentials that allow transaction syncing — are stored server-side only, are restricted at the column level, and are never readable by the client application. Consumer financial data accessed through the Service is never sold, licensed, or shared for any party's independent benefit; it is used only to deliver the features the account holder has requested.

5. Infrastructure and Network Security

The Service runs on Vercel serverless infrastructure for application and API hosting, with a Supabase-managed PostgreSQL database for data, authentication, and file storage. The database is not publicly exposed for direct access; application and API code reach it through controlled, authenticated server-side connections.

All secrets and credentials are stored in protected environment variables on the hosting and database platforms. Secrets are never committed to source control and are never included in client-side bundles.

6. Secure Development and Change Management

All source code is maintained under Git version control, providing a full, auditable history of changes. Production deployments are automated from the main branch and require a successful build to ship, preventing broken or unverified code from reaching production. Third-party dependencies are sourced from trusted registries and kept updated to incorporate security fixes.

7. Third-Party and Vendor Management

Wireway relies on a small set of vetted subprocessors, each accessed only for its specific function and provided only the data necessary to perform it:

  • Plaid — secure connection to users' bank and card accounts for transaction data.
  • Stripe — payment processing and payout to contractors.
  • Supabase — authentication, database, and file storage.
  • Vercel — application and API hosting and deployment.

We access only the data each subprocessor needs to deliver its function, and we rely on these providers' own security and compliance programs as part of our overall control environment.

8. Logging and Monitoring

Platform and application logs are retained and reviewed to detect errors and security-relevant events, and to support investigation and troubleshooting. Authentication events — such as sign-ins and credential changes — are handled and recorded by our managed authentication provider, which provides the underlying account-security controls for the Service.

9. Incident Response

Suspected or confirmed security incidents are investigated promptly by the Information Security Lead. Our response prioritizes containment first — for example, rotating credentials and revoking access — to limit any potential impact before remediation.

If an incident involves data accessed through Plaid, we will notify Plaid in accordance with our agreement and notify affected users without undue delay. In all cases, we work to identify and remediate the root cause and to apply controls that reduce the likelihood of recurrence.

10. Data Retention and User Rights

Users remain in control of their data. A user can export all of their own data from the Service at any time. Upon a verified deletion request, we remove the user's data — including financial data and any stored access tokens. When a user disconnects a linked financial institution or deletes their account, the associated Plaid access tokens are revoked so that no further data can be accessed.

11. Policy Review

This policy is owned by the Information Security Lead and is reviewed at least annually, and after any material change or security incident, to ensure it remains accurate and effective. This version is effective June 27, 2026.
